Technical Reference

Architecture, internals, and developer information.

Architecture Overview

Atari800MacX is built as an Objective-C + Swift + C application. The C emulation core comes from the Atari800 open-source project. The macOS UI layer uses AppKit (Objective-C) with a modern SwiftUI preferences panel. Rendering uses Metal via an MTKView subclass, and audio/HID input use SDL2.

Key Design Decisions

  • ObjC + Swift coexistence — the bridging header (Atari800MacX-Bridging-Header.h) exposes key ObjC classes to Swift. Swift code uses ObservableObject/@Published (not @Observable) for macOS 13+ compatibility.
  • Metal rendering — replaced the original SDL renderer. EmulatorMetalView (MTKView subclass) draws emulator output via a Metal render pipeline with custom shaders.
  • SDL2 for audio & HID — SDL2 v2.0.14 (universal binary) handles audio output and game controller input. The SDL window/surface/texture path has been removed.
  • ARC enabled — Objective-C code uses ARC (CLANG_ENABLE_OBJC_ARC = YES) across all build configurations.
  • Carbon weak-linked — -weak_framework Carbon in OTHER_LDFLAGS for TIS keyboard layout functions, with NULL guards for graceful degradation.

Emulated Hardware

CPU — MOS 6502C

  • Full 6502 instruction set with cycle-accurate timing option
  • All official and undocumented opcodes supported
  • Cycle-stealing by ANTIC DMA accurately emulated

ANTIC — Display Controller

  • All 16 display modes (modes 2–F, plus blank lines and jump instructions)
  • Display List Interrupts (DLI)
  • Fine and coarse horizontal/vertical scrolling
  • Player/missile DMA
  • VSCROL and HSCROL registers
  • WSYNC cycle-exact wait for horizontal sync

GTIA — Graphics Controller

  • 4 players + 4 missiles with configurable widths (1×, 2×, 4×)
  • 8 priority modes for player/playfield layering
  • GTIA modes 9 (16 luminances), 10 (9 colors), and 11 (16 hues)
  • Collision detection registers
  • Console key inputs (Start, Select, Option)

POKEY — Sound & I/O

  • 4 audio channels with independent frequency and control
  • Polynomial counter noise generation (4-bit, 5-bit, 17-bit, 9-bit)
  • High-pass filter (channels 1+3, 2+4 clocking)
  • 16-bit timer mode (channels 1+2 or 3+4 joined)
  • Serial I/O (SIO) bus controller
  • Keyboard scanning and debounce
  • IRQ generation (timer, serial, keyboard, break key)
  • Dual POKEY (stereo) support

PIA — Peripheral Interface

  • Port A: joystick inputs (2 joystick ports, 4 bits each)
  • Port B: memory bank control (XL/XE), self-test ROM enable, BASIC ROM enable
  • IRQ on peripheral connect/disconnect

Rendering Pipeline

Frame Production

  1. The ANTIC/GTIA emulation writes indexed color values to Screen_atari (the standard Atari800 frame buffer).
  2. Atari_DisplayScreen() converts indexed values to BGRA pixels using MetalPalette32[256] and writes to MetalFrameBuffer.
  3. For 80-column modes, the appropriate display function (DisplayWithoutScaling16bppAF80, etc.) writes directly to MetalFrameBuffer.
  4. Mac_MetalPresent() triggers the Metal view to upload and render the frame.

Palette Generation

CalcPalette() builds MetalPalette32 as an array of 256 BGRA8 values. Each entry is computed from the Atari color table with the alpha channel set to fully opaque (0xFF).

ATR Boot Sector Modes

Double-density ATR images support three boot sector layouts for the first 3 sectors:

  • Logical — 128 bytes per boot sector (standard)
  • Physical — 256 bytes per sector, first half used, second half zero-padded
  • SIO2PC — 3×128 data bytes followed by 3×128 unused bytes

Built-in Altirra ROMs

When original ROMs are not available, Atari800MacX uses built-in replacement ROMs from the Altirra project by Avery Lee:

  • Altirra OS (800) — replaces Atari 800 OS ROM
  • Altirra OS (XL) — replaces XL/XE OS ROM
  • Altirra BASIC — replaces Atari BASIC ROM
  • Altirra 5200 BIOS — replaces 5200 BIOS

These are compiled directly into the binary as C arrays and provide excellent compatibility with most software.

Expansion Hardware Internals

SIDE2 Registers ($D5xx)

The SIDE2 cartridge maps its registers at $D500–$D5FF. Key functions include CompactFlash I/O, flash ROM banking, and SDX/Loader mode selection.

FunctionDescription
SIDE2_D5GetByte/D5PutByteRegister read/write at $D5xx
SIDE2_Add_Block_DeviceMount CompactFlash image
SIDE2_SDX_Switch_ChangeToggle SDX/Loader mode
SIDE2_Flash_Read/WriteDirect flash ROM access

Ultimate 1MB Registers ($D1xx/$D3xx/$D5xx/$D6xx–$D7xx)

The Ultimate 1MB expansion uses registers across multiple address ranges:

AddressFunction
$D1xxPrimary configuration registers
$D1FFSpecial control register
$D3xxBank selection and OS switching
$D5xxFlash ROM banking
$D6xx–$D7xxVBXE register passthrough (when VBXE enabled)

VBXE Registers ($D640 or $D740)

The Video Board XE maps 256 bytes of registers at a configurable base address. Key registers:

OffsetRegisterDescription
$08VIDEO_CONTROLBit 7: XDL enable
$2BBLITTER_STARTWrite $7F to trigger blit operation
$00–$07XDL pointersExtended Display List base address and configuration
$10–$1FBlitter registersSource/dest addresses, dimensions, control
$40–$7FPalette21-bit RGB palette (256 entries)

VBXE uses MEMAC windows for CPU access to its 512 KB VRAM. Register dispatch goes through PBI_D6GetByte/PutByte ($D640 base) or PBI_D7GetByte/PutByte ($D740 base).
