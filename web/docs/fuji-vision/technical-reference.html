Technical Reference

Architecture, internals, and developer information.

Architecture Overview

Fuji-Vision is built as a pure Swift + C application with no Objective-C layer. The C emulation core comes from the Atari800 open-source project, compiled directly into the visionOS app via a bridging header.

Layer Diagram

┌──────────────────────────────────┐
│   SwiftUI Views & App Lifecycle  │  Swift / visionOS SDK
├──────────────────────────────────┤
│   EmulatorRenderer (Metal)       │  MTKView + Metal shaders
│   AudioManager (AVAudioEngine)   │  AVAudioSourceNode pull model
│   InputManager (GameController)  │  GCController framework
│   VisibilityCompositor           │  Data processor (pure Swift)
├──────────────────────────────────┤
│   Platform Bridge (C)            │  atari_vision.c, stubs
├──────────────────────────────────┤
│   Atari800 C Core                │  ~57 source files from atari800
│   Altirra ROMs                   │  Built-in OS/BASIC/5200 ROMs
└──────────────────────────────────┘

Key Design Decisions

  • No ObjC — visionOS uses UIKit (not AppKit), and all platform glue is done through a C bridging header, avoiding the need for an Objective-C bridge.
  • Dedicated emulation thread — The C core runs on its own pthread, decoupled from the main/render thread. Frame data is copied to a shared buffer and uploaded to a Metal texture on the main thread.
  • Pull-model audio — Audio samples are produced by the emulation thread into a lock-free ring buffer and pulled by AVAudioSourceNode's render callback, avoiding audio glitches from thread contention.
  • Built-in ROMs — Altirra OS, BASIC, and 5200 ROMs are compiled directly into the binary as C arrays, so no external ROM files are required.

Rendering Pipeline

Frame Production (C Core)

  1. The emulation core writes pixel data to Screen_atari (the standard Atari800 frame buffer).
  2. PLATFORM_DisplayScreen() calls Atari800Core_GetFrameBuffer(&w, &h), which converts Screen_atari into an RGBA8888 pixel buffer (typically 384×240 pixels) using the active color lookup table.
  3. A registered frame callback copies the pixel data and notifies the renderer.

Frame Upload (Metal)

  1. EmulatorRenderer.uploadFrame() receives the RGBA pixel data on the main thread.
  2. Pixels are uploaded to an MTLTexture with pixel format .rgba8Unorm.
  3. Before returning, uploadFrame() also calls compositor.analyzeFrame() for background detection (if auto-detect is enabled).

Shader Pipeline

The render pipeline uses a vertex shader (emulatorVertex) and fragment shader (emulatorFragment) loaded from the app's default Metal library.

MTKView Configuration

For visionOS pass-through transparency, the MTKView is configured with:

  • isOpaque = false
  • clearColor alpha = 0 (fully transparent background)
  • framebufferOnly = false (required for compositor frame reads)
  • layer.isOpaque = false

Texture Sampling

Two sampler states are maintained: nearest-neighbor (sharp pixels) and linear (bilinear filtering). The active sampler is selected by the FV_LinearFilter setting. Both use clamp-to-edge addressing.

Buffer Management

The ring buffer uses a single-producer, single-consumer (SPSC) lock-free design. The emulation thread writes samples; the audio render callback reads them.

  • Underrun — If the buffer runs dry, the render callback fills with silence (zero samples). The emulation speed is adjusted to 105% to catch up.
  • Overrun — If the buffer exceeds its fill target by more than 12.5%, emulation speed is reduced to 95% to let audio drain.

Volume Control

Volume scaling is applied in Sound_Update() on the emulation thread. Each int16_t sample is multiplied by the sound_volume factor (0.0–1.0) before being written to the ring buffer.

Input System

GCController Framework

Fuji-Vision uses Apple's GameController framework to interface with Bluetooth controllers. Controller discovery starts on app launch via GCController.startWirelessControllerDiscovery(). Only the first connected controller is tracked.

Joystick Interface

Joystick state is written directly to the Atari800 core's hardware registers:

  • PIA_PORT_input[0] — joystick direction bits (port 0)
  • GTIA_TRIG[0] — fire button state (0 = pressed, 1 = released)

INPUT_Frame() is a no-op stub; input state is updated directly by the controller handlers rather than being polled per frame.

Visibility Compositor

VisibilityCompositor is a pure data processor—it holds no Metal objects and produces a FragParams struct consumed by the renderer each frame.

Alpha Transitions

All mode changes animate via linear interpolation over 0.3 seconds (lerp duration). The update(deltaTime:) method is called once per Metal draw call, advancing the current alpha toward the target at approximately 3.33 alpha units per second.

Background Detection Algorithm

  1. Guard — Only runs when autoDetectEnabled == true and detectedColorLocked == false.
  2. Interval — Runs every 30 frames (~0.5 seconds at 60 fps).
  3. Sampling — Reads border pixels from the top row, bottom row, left column, and right column of the RGBA frame buffer, sampling every 4th pixel.
  4. Quantization — Each sampled pixel's RGB channels are quantized to 4-bit precision (16 levels per channel) and combined into a 12-bit key: (qR << 8) | (qG << 4) | qB.
  5. Histogram — A frequency histogram is built from the quantized keys. The dominant color is the key with the highest count.
  6. Dequantization — The winning key is converted back to floating-point RGB by centering within each bucket: (nibble + 0.5) / 16.0.
  7. Hysteresis — The detected color must remain stable for 3 consecutive detection cycles before it is adopted as the new key color. Stability is measured as simd_distance(detected, candidate) < threshold.
  8. Sensitivity mapping — The sensitivity slider (0.0–1.0) maps to a distance threshold via: threshold = 0.1 × (1.0 - sensitivity) + 0.02.
    • Sensitivity 0.0 → threshold 0.12 (most tolerant)
    • Sensitivity 0.5 → threshold 0.07
    • Sensitivity 1.0 → threshold 0.02 (most strict)
  9. Auto-enable — When hysteresis is satisfied and chroma key is currently off, the compositor automatically enables keying.

Build Information

The Xcode project is generated from project.yml using XcodeGen:

/opt/homebrew/bin/xcodegen generate

Run this from the apps/fuji-vision/ directory whenever project.yml changes.

xcodebuild -scheme FujiVision \
  -destination 'generic/platform=xrsimulator' \

Header Search Path Ordering

The FujiVision/Platform/ directory is listed first in the header search paths. This ensures the visionOS-specific config.h (which defines VISIONOS 1, ATARI800MACX, and MACOSX) takes precedence over the fuji-foundation macOS config.h.

C Core Sources

Approximately 57 C source files from the Atari800 core are included, plus 5 platform bridge files and 4 Altirra ROM data files. The following files are excluded from the visionOS build due to SDL, IOKit, or other macOS-specific dependencies:

  • preferences_c.c (requires SDL)
  • capslock.c (requires IOKit)
  • rdevice.c (requires R_SERIAL/R_NETWORK)
  • input.c, screen.c, colours.c, sound.c, monitor.c (replaced by platform-specific implementations)
  • atari_mac_sdl.c (SDL main loop)

Replacement files: atari_vision.c, preferences_vision.c, capslock_vision.c, vision_stubs.c, and mac_monitor.c.
